User flows

Screenshots from the live app

Real screens from flick-uxmaxx.vercel.app, not mockups.

1

Login — no wallet talk

Email or Google, nothing else. Behind the scenes, Particle Auth Core creates an embedded MPC wallet tied to that account — the user never sees a seed phrase or "connect wallet" prompt.

Client calls connect() → on success, POST /api/users registers particle_user_id in Supabase.
2

Home — one balance, any chain

Base and Arbitrum balances converge into a single number. The public ticker at the bottom polls a CORS-open, PII-free aggregate endpoint every 5 seconds.

GET /api/stats/flicks-count — service-role read, exposes only a count.
3

Send — the real transaction

Amount, recipient, optional emoji/message, optional group split. Tapping "Flick" triggers a real USDC transfer from the sender's wallet to the escrow address — signed in the browser, not simulated.

Client: useEthereum().sendTransaction() (viem-encoded ERC-20 transfer) → server verifies the tx before creating the claim.
4

Claim — gift-card mental model

The recipient sees an amount and a message, not a wallet address. On claim, the escrow pays them directly on-chain — the claim only flips to "claimed" if that payment actually succeeds.

POST /api/claims/[token] → sendUsdcFromEscrow() → DB updated only after on-chain success.
Security model

What's actually enforced

  • ✓Server-side session verification — every mutating request calls Particle's getUserInfo RPC to confirm the submitted user ID and token belong to a real, currently valid session. Verified live: fake/missing tokens get a hard 401.
  • ✓Deny-by-default Row Level Security on every table. Verified live with the public key: reads return empty arrays, writes get rejected with an explicit RLS violation. All real reads/writes go through server routes using the service role.
  • ✓On-chain verified transfers — the backend never trusts a client-submitted transaction hash. It fetches the real receipt from a public RPC, decodes the ERC-20 Transfer event, and checks the recipient and amount before creating or completing a claim.
  • ✓Fail-closed payouts — if the on-chain payout to a recipient fails, the claim stays "pending" instead of being marked claimed. No double-pay, no silent fund loss.
  • ✓No secrets in code — private keys and service-role keys live only in Vercel environment variables, never committed, never sent to the client.
  • ✓Non-guessable claim links — claim_token is 128 bits of cryptographic randomness, not a sequential ID.

Known limitations + roadmap

What's next, honestly

Universal Account is mainnet-only today

Particle's Universal Account SDK doesn't support testnet chain IDs yet. The testnet demo uses a transparent escrow relay instead — same UX, real transactions, different plumbing under the hood until testnet support lands.

Wallet-connect for existing wallets

Today every login creates a fresh Particle wallet. A real product needs to let users who already hold funds elsewhere connect their existing wallet instead of starting from zero.

Real balance reads

The home screen balance is currently a fixed example value — swapping in a live Universal Account balance read is the next real integration step.

Rate limiting

No request-rate limiting yet on the API routes — fine for a hackathon demo, necessary before any real traffic.
